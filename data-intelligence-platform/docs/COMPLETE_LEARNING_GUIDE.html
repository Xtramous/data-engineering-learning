The Problem & Solution

The Real Problem

Imagine you're a data engineer at a company with:

  • Hundreds of data tables across multiple systems
  • Dozens of dashboards and reports
  • Complex ETL pipelines (Airflow DAGs)
  • Data flowing from source → transformation → consumption
  • Multiple teams managing different assets

Now you need to answer questions like:

  • ❓ "What feeds the customer_fact table?"
  • ❓ "Which dashboards use sales data?"
  • ❓ "Who owns this table and when was it last updated?"
  • ❓ "Can I safely modify this column?"
  • ❓ "What's the impact if this table goes down?"

Traditional solutions: Ask people (slow), search Excel sheets (unreliable), read docs (outdated)

Our Solution: Data Intelligence Platform

We built an automated system that:

  • ✅ Discovers all data assets automatically
  • ✅ Maps complete data lineage (source → target)
  • ✅ Makes data searchable and discoverable
  • ✅ Provides an AI assistant to answer questions
  • ✅ Runs completely locally (no cloud bills, complete privacy)
"This is inspired by Select Star, which is a commercial metadata discovery tool. We built a simplified, local version that you can run on your laptop."

PostgreSQL: The Data Warehouse

What is PostgreSQL?

PostgreSQL is a relational database management system. Think of it as a collection of organized tables with columns and rows, where you can find anything using SQL queries.

Why PostgreSQL for This Project?

  • ✅ ACID Compliance: Your data is always consistent
  • ✅ Complex Queries: Can JOIN multiple tables and aggregate
  • ✅ Indexing: Fast lookups on specific columns
  • ✅ Open Source: Free, no licensing costs
  • ✅ Reliable: Been around for 20+ years

Our Tables

┌─────────────────────────────┐ │ public schema │ ├─────────────────────────────┤ │ customer_dim │ 10,000 rows - Who are the customers? ├─────────────────────────────┤ │ product_dim │ 5,000 rows - What products do we have? ├─────────────────────────────┤ │ sales_fact │ 500,000 rows - What sold when? ├─────────────────────────────┤ │ order_fact │ 100,000 rows - Orders and statuses ├─────────────────────────────┤ │ inventory_fact │ 50,000 rows - Stock levels └─────────────────────────────┘ ┌─────────────────────────────┐ │ metadata schema │ ├─────────────────────────────┤ │ table_metadata │ Metadata about tables ├─────────────────────────────┤ │ column_metadata │ Metadata about columns └─────────────────────────────┘

Table: sales_fact

Purpose: Stores transaction-level sales data

Column Type Meaning

product_id INTEGER (FK) References product_dim
sales_date DATE When the sale happened

Relationships (Foreign Keys)

customer_dim ──→ sales_fact (many customers can have many sales) product_dim ──→ sales_fact (many products can have many sales) customer_dim ──→ order_fact (many customers can have many orders) product_dim ──→ inventory_fact (one product can have inventory in many locations)

SQL Example: Common Queries

-- Get total revenue by customer SELECT c.customer_name, SUM(s.total_amount) as total_revenue, COUNT(s.sales_id) as num_sales FROM customer_dim c JOIN sales_fact s ON c.customer_id = s.customer_id GROUP BY c.customer_id, c.customer_name ORDER BY total_revenue DESC; -- Find top selling products SELECT p.product_name, SUM(s.quantity) as units_sold, SUM(s.total_amount) as revenue FROM product_dim p JOIN sales_fact s ON p.product_id = s.product_id GROUP BY p.product_id, p.product_name ORDER BY units_sold DESC LIMIT 10;

Indexing Strategy

Indexes speed up queries by organizing data like a book index:

Index On Column Why
idx_customer_email customer_dim.email Fast lookup by email
idx_sales_date sales_fact.sales_date Fast date range queries
idx_sales_customer sales_fact.customer_id Fast join with customer_dim
idx_sales_product sales_fact.product_id Fast join with product_dim

💡 Key Concept: Normalization

Our design uses dimension and fact tables:

  • Dimensions: customer_dim, product_dim (describes what/who)
  • Facts: sales_fact, order_fact (describes what happened)

This pattern makes data easy to understand, query, and maintain.

Neo4j: The Lineage Graph Database

What is Neo4j?

Neo4j is a graph database. Instead of tables, it stores nodes (things) and relationships (connections between things).

Analogy: If PostgreSQL is a spreadsheet, Neo4j is a network diagram where boxes are connected with lines.

Why Neo4j for Lineage?

  • ✅ Natural Fit: Data lineage IS a graph
  • ✅ Fast Traversal: Following relationships is blazingly fast
  • ✅ Pattern Matching: Find complex patterns easily
  • ✅ Visual: Natural to render as diagrams

Core Concepts

🔹 Nodes

Entities in the graph. Each has:

  • Label (type): Table, Column, Dashboard, Job
  • Properties: name, owner, description, etc.

🔗 Relationships

Connections between nodes. Each has:

  • Type: FEEDS_INTO, USED_BY, OWNS
  • Properties: description, metadata

Our Graph Structure

(Table:customer_dim) │ ├─[FEEDS_INTO]──→ (Table:sales_fact) │ │ │ ├─[USED_BY]──→ (Dashboard:sales_dashboard) │ │ │ └─[USED_BY]──→ (Dashboard:financial_dashboard) │ └─[OWNS]──→ (Team:analytics-team) (Table:product_dim) │ ├─[FEEDS_INTO]──→ (Table:sales_fact) │ └─[OWNS]──→ (Team:analytics-team) (Job:sales_ingest_dag) │ └─[PROCESSES]──→ (Table:sales_fact)

Node Types

Node Type Example Properties
Table customer_dim name, schema, owner, row_count, description
Column customer_id name, type, nullable, table_name
Dashboard sales_dashboard id, name, owner, description
Job/DAG customer_etl name, owner, schedule, description

Relationship Types

Relationship From To Meaning
FEEDS_INTO Table/Job Table Data flows from source to target
USED_BY Table Dashboard/Report Table is queried by dashboard
DEPENDS_ON Job/Task Table Job requires this table
OWNS Team/Person Asset Who is responsible

Cypher Query Language

Cypher is Neo4j's query language. It looks like ASCII art of the graph!

-- Find all tables that feed into customer_fact MATCH (source:Table)-[r:FEEDS_INTO]->(target:Table {name: 'customer_fact'}) RETURN source.name, r.description -- Find all downstream impact (what breaks if I drop this table?) MATCH path = (table:Table {name: 'customer_dim'})-[r:FEEDS_INTO*1..3]->(downstream) RETURN path -- Find who owns what MATCH (team)-[r:OWNS]->(asset) WHERE asset.name = 'customer_dim' RETURN team.name, asset.name -- Get lineage depth (how many hops?) MATCH path = (source:Table)-[r:FEEDS_INTO*]->(target:Table) WHERE source.name = 'raw_customers' AND target.name = 'sales_fact' RETURN length(path) as hops

Example: Complete Lineage for sales_fact

UPSTREAM (what feeds into it): raw_customers ──→ customer_dim ↘ ↙ ╲ ╱ ╲ ╱ ╲ ╱ ╲ ╱ sales_fact DOWNSTREAM (what uses it): sales_fact ──→ sales_dashboard ├──→ financial_dashboard ├──→ customer_dashboard └──→ revenue_metric (in other reports) IMPACT ANALYSIS: If sales_fact goes down → All downstream break If raw_customers goes down → sales_fact incomplete If customer_dim goes down → sales_fact can't be enriched

🎯 Power of Graphs

With relational database, finding "all downstream impact" would require complex SQL with multiple JOINs. With Neo4j, it's one simple query. That's the power of graph databases for lineage!

OpenSearch: Full-Text Search & Embeddings

What is OpenSearch?

OpenSearch is a search and analytics engine. It's like Google for your data—you type a query and it finds relevant results, ranked by relevance.

Why OpenSearch?

  • ✅ Full-Text Search: Find "customer" in any document field
  • ✅ Vector Search: Find semantically similar items
  • ✅ Relevance Ranking: Best results first
  • ✅ Real-Time Indexing: Fast updates
  • ✅ Open Source: Fork of Elasticsearch

Two Types of Search

🔤 Full-Text Search

What: Keyword matching

Query: "sales"

Returns: Documents containing "sales", "Sales", "SALES"

Example: Find all tables about sales

🧠 Semantic Search

What: Meaning matching

Query: "revenue"

Returns: Tables about income, sales, earnings, etc.

Example: Find metrics about money coming in

How Search Works

1. User enters: "customer data" │ ▼ 2. Split into terms: ["customer", "data"] │ ▼ 3. Search for "customer" in inverted index (inverted index maps word → document) │ ▼ 4. Search for "data" in inverted index │ ▼ 5. Combine results (docs with both terms score highest) │ ▼ 6. Rank by relevance score (BM25 algorithm) (matches in title > matches in description) │ ▼ 7. Return top 20 results with scores │ ▼ 8. Display to user

Our Indices (Search Collections)

Index Contains Searchable Fields
tables_index Table metadata name, description, schema, owner
columns_index Column metadata name, type, description, table_name
dashboards_index Dashboard metadata name, description, owner, tables
documentation_index Markdown documentation content (full text)

Index Document Example

{ "id": "customer_dim", "name": "customer_dim", "schema": "public", "owner": "analytics-team", "description": "Customer dimension table with customer details and signup information", "row_count": 10000, "columns": ["customer_id", "customer_name", "email", "phone", "city", "country", "signup_date", "is_active"], "embedding": [0.123, 0.456, 0.789, ...], // 1024-dimensional vector "tags": ["customer", "dimension", "master-data"], "created_at": "2024-01-15", "updated_at": "2024-06-15" }

Vector Embeddings

Embedding: A numerical representation of meaning.

The sentence "customer_dim stores customer information" gets converted to a vector of 1024 numbers that represents its meaning.

Text: "customer dimension with customer details" │ ▼ Ollama (LLM) processes text │ ▼ Returns 1024-dimensional vector: [0.123, 0.456, 0.789, ..., 0.234] │ ▼ Stored in OpenSearch │ ▼ When user searches for "who has customer info?" ▼ Convert query to vector │ ▼ Find vectors closest to query vector │ ▼ Return semantically similar results

Search in Action

-- Full-text search (backend calls this) GET /tables_index/_search { "query": { "multi_match": { "query": "customer", "fields": ["name", "description", "owner"] } } } -- Vector similarity search GET /tables_index/_search { "query": { "knn": { "embedding": { "vector": [0.123, 0.456, ...], "k": 5 } } } } -- Combined search (best of both) GET /tables_index/_search { "query": { "bool": { "should": [ {"match": {"description": "customer"}}, {"knn": {"embedding": {...}}} ] } } }

💡 Why Both Types?

Full-text: Fast, exact, works well for exact matches

Semantic: Slow but smart, understands meaning

Using both gives you the best of both worlds!

Ollama: Local LLM

What is Ollama?

Ollama is a tool to run Large Language Models (LLMs) locally on your computer. It's like having ChatGPT running on your machine, without sending data to the cloud.

Why Ollama?

  • ✅ Private: All data stays on your machine
  • ✅ Free: No API costs
  • ✅ Fast: No network latency
  • ✅ Offline: Works without internet
  • ✅ Reliable: Your infrastructure, your control

LLM (Large Language Model) Basics

An LLM is an AI that:

  1. Takes text as input
  2. Understands what you're asking
  3. Generates relevant text as output

Example:

  • Input: "What feeds customer_fact?"
  • LLM thinks about question and context
  • Output: "customer_fact is fed by raw_customers table, which is extracted daily via the customer_etl DAG..."

Two Uses in Our System

1️⃣ Embeddings

What: Convert text to numbers

Input: "Customer dimension table"

Output: [0.123, 0.456, ..., 0.789] (1024 numbers)

Use: Semantic search in OpenSearch

2️⃣ Chat

What: Generate human-readable responses

Input: "What feeds customer_fact?" + context data

Output: Natural language explanation

Use: AI Assistant in chat page

Llama 3: Our Chosen Model

Llama 3 is Meta's open-source LLM. We chose it because:

  • ✅ Open Source: Completely free and modifiable
  • ✅ Good Quality: Performs well on various tasks
  • ✅ Reasonable Size: ~4GB (fits on most laptops)
  • ✅ Well Maintained: Regular updates

How Ollama Works

1. Start Ollama service on port 11434 │ ▼ 2. Backend requests embedding │ ▼ 3. POST http://localhost:11434/api/embeddings { "model": "llama3", "prompt": "customer dimension table" } │ ▼ 4. Ollama loads Llama 3 model to GPU/CPU │ ▼ 5. Model processes text │ ▼ 6. Returns JSON with embedding vector │ ▼ 7. Backend indexes vector in OpenSearch

Example: Chat Flow

User asks: "What feeds customer_fact?" │ ▼ Backend receives question │ ▼ Backend queries Neo4j: MATCH (source:Table)-[r:FEEDS_INTO]->(target:Table {name: 'customer_fact'}) RETURN source.name, r.description │ ▼ Neo4j returns: [ {source: "raw_customers", description: "Raw customer data"}, {source: "raw_updates", description: "Customer updates"} ] │ ▼ Backend crafts context prompt: "Based on the data lineage, customer_fact is fed by: - raw_customers: Raw customer data - raw_updates: Customer updates Explain this to the user." │ ▼ Backend calls Ollama: POST /api/generate { "model": "llama3", "prompt": "Based on the data lineage, customer_fact is fed by...", "stream": false } │ ▼ Ollama generates response: "customer_fact is populated from two main sources... The raw_customers table contains initial customer information... The raw_updates table contains... This data is processed daily by the customer_etl DAG..." │ ▼ Backend returns response to Frontend │ ▼ Frontend displays answer to user

⚠️ LLM Limitations

LLMs are powerful but not perfect:

  • Can "hallucinate" (make up facts)
  • May give inconsistent answers
  • Need good context to work well
  • Slower than traditional code

Solution: Always provide factual context from databases. Let the LLM explain, not discover.

FastAPI: Backend Framework

What is FastAPI?

FastAPI is a modern Python framework for building APIs (Application Programming Interfaces). It's the glue between your databases and the frontend.

Why FastAPI?

  • ✅ Fast: Built on async/await for high performance
  • ✅ Modern: Uses Python 3.7+ features
  • ✅ Type Safe: Built-in type hints and validation
  • ✅ Auto Documentation: Swagger UI out of the box
  • ✅ Easy to Learn: Clear, intuitive API
